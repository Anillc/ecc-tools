# Design: CTS optimization critical-branch char sizing

## Flow Boundary

The optimization stage is a peer of synthesis and instantiation:

```text
readData()
  -> Synthesis::run(clock_layout, char_library)
  -> DESIGN_INST.rebuildClockDAG()
  -> Optimization::run(clock_layout, char_library)
  -> DESIGN_INST.rebuildClockDAG()
  -> Instantiation::run()
  -> Evaluation::run()
```

`Flow` owns a shared `CharacterizationLibrary` so synthesis and optimization can reuse H-tree basic char results. The optimization stage may ensure the shared library once before solving if synthesis did not leave usable char data.

## Layer Responsibilities

`flow/optimization` owns all committed-design access:

- reads `CONFIG_INST` target skew and max cap,
- reads `DESIGN_INST` clocks and `ClockDAG`,
- reads `ClockLayout`,
- queries `STA_ADAPTER_INST` for pin cap, cell area, output cap limit, and wire cap,
- builds `CharTimingLookup`,
- builds explicit `TreeSizingProblem`,
- applies accepted sizing mutations to `Inst` and matching `ClockLayout` records,
- emits concise optimization reports.

`module/buffer_sizing` is pure algorithm code:

- evaluates assignments from explicit problem data,
- calls `CharTimingLookup`,
- checks explicit cap limits,
- computes sink arrivals and skew,
- runs LCA-guided critical branch sizing,
- returns accepted decisions and summary metrics.

It must not read `CONFIG_INST`, `DESIGN_INST`, `STA_ADAPTER_INST`, iDB, iSTA, or `ClockLayout`.

`utils/graph` owns generic rooted-tree LCA utilities with no CTS dependencies.

## CharTimingLookup

`CharTimingLookup` is the only timing query layer used by the solver. It is not a fitting model. It standardizes access to raw H-tree basic char data.

Responsibilities:

- map requested branch length to the char wirelength lattice,
- distinguish pure-wire arcs and terminal-buffer arcs,
- find the four surrounding slew/cap lattice points,
- bilinearly interpolate delay and output slew,
- report boundary clamping and missing coverage,
- optionally provide wire/cell split for diagnostics.

It must not:

- fit delay models,
- query Liberty timing,
- query STA,
- make optimization decisions,
- read global design/config state.

Do not introduce a separate `TreeCharTimingOracle` abstraction. If a tree-level timing component is useful during implementation, keep it as an evaluator/helper inside `TreeBufferSizing` with an explicit name such as `TreeTimingEvaluator`. The word "oracle" is avoided because this layer is not an independent timing authority; it only propagates assignment-specific tree timing using `CharTimingLookup` results.

## Timing Calculation Chain

Raw char is necessary but not sufficient for optimization timing. The chain is:

1. `CharacterizationLibrary` owns the reusable H-tree basic char generated by `CharBuilder`.
2. `CharTimingLookup` indexes those raw samples by length lattice, pattern kind, buffer master, input slew, and load cap, then returns interpolated segment delay/output slew.
3. `flow/optimization` converts the synthesized CTS result into an explicit tree problem: parent/child topology, routed branch length, fixed load cap, sink pin cap, candidate buffer input caps, area, and max-cap limits.
4. `TreeBufferSizing` evaluates one sizing assignment by computing assignment-dependent net loads, calling `CharTimingLookup` for each tree arc in topological order, propagating arrival/slew from the clock source to every sink, and reporting global min/max sink arrival.
5. The critical-branch sizing loop uses the full-tree evaluation result to decide whether a trial upsize strictly improves skew without introducing cap violations.

The implementation must not calculate Liberty delay, Elmore/RCTree timing, or ad-hoc slew formulas inside optimization. Length discretization and segment delay/slew must come from char lookup; tree-level arrival propagation is only additive stitching of those char query results across the actual synthesized tree.

## Tree Model

The flow builds a rooted tree with nodes for:

- clock source,
- reachable CTS buffers,
- terminal clock sinks.

Each buffer has:

- current master,
- current-or-larger candidate choices,
- input pin cap,
- area,
- output cap limit,
- parent net,
- output nets.

Each net has:

- driver node,
- child buffer loads,
- terminal sink loads,
- fixed loads,
- total wire cap,
- baseline load cap,
- max cap.

The module computes assignment-dependent net load:

```text
wire cap + fixed load cap + selected child buffer input caps + terminal sink pin caps
```

## Solver

The solver loops:

1. Evaluate the current assignment.
2. Stop if skew is at or below target.
3. Find min/max sink nodes and their LCA.
4. Extract the LCA-to-max-sink branch path.
5. Scan buffer candidates from LCA toward max sink.
6. For each buffer, try legal upsize choices.
7. Evaluate full-tree timing for each trial.
8. Accept the best candidate that strictly improves global skew and has no new cap violation.
9. Repeat until target met, no candidate improves, or budget reached.

Tie-breaks among improving candidates:

1. larger skew improvement,
2. smaller area delta,
3. smaller drive-rank step,
4. closer to LCA in the root-to-sink order.

Max sink identity may change. The accepted mutation criterion is full-tree skew improvement, not preserving the sink identity.

## Reporting

Default logs contain one concise optimization summary plus mutation transition distribution. Large per-buffer tables and full path dumps are not emitted by default.

## Rollback

Accepted mutations are applied only after the solver returns a successful solution. If applying any mutation fails, flow reports failure and does not continue to instantiation.
