## Current Runtime Report Flow

Top-level report behavior:

- `Setup::run` opens `CONFIG_INST.get_log_file()` as `cts.log` through `SCHEMA_WRITER_INST.open(...)` and writes the run title plus `Run Context`.
- `Setup::emitRuntimeSetup` writes runtime paths, runtime configuration, and routing/wire RC.
- `Flow::runCTS` starts total runtime tracking and a top-level stage named `CTS Clock tree synthesis API flow`.
- `Flow::readData` emits `## Input Overview` and `### Clock Data`.
- `Synthesis::run` emits `## Synthesis Overview`.
- `Instantiation::run` emits `### iDB Conversion`.
- `Evaluation::run` emits `## Evaluation Overview` and `### Final Evaluation`.
- `Flow::emitKeyResults` emits `## Run Results` and `CTS Key Results`.
- `Report::run` emits `## Report Overview`, report mode, visualization/GDS report tables, report runtime, and report-stage summary.

Schema helper behavior:

- `SchemaWriter::emitTable` and `emitKeyValueTable` write directly to the active schema file.
- Free functions `schema::EmitTable` and `schema::EmitKeyValueTable` dual-write: they emit the same table to `LOG_INFO` and to `SCHEMA_WRITER_INST`.
- `SchemaWriter::StageScope` writes `Context` tables when constructed with start fields and writes `Summary` tables on `finished`, `failed`, or `skip`.
- `StageScope` also emits console-only start/finish markers through `LOG_INFO`.

This means `cts.log` is currently both a human report and an execution trace whenever internal helpers use `beginStage(...)` with start and finish fields.

## Redundancy Patterns

### Per-depth HTree repetition

For four evaluated depth candidates, the sample repeats the same six tables:

- `HTreeDepth Build pattern frontier Context`
- `HTreeDepth Apply root-driver compensation Context`
- `HTreeDepth Apply root-driver compensation Summary`
- `HTreeDepth Build pattern frontier Summary`
- `HTreeDepth Filter sink-load region Context`
- `HTreeDepth Filter sink-load region Summary`

That is 24 tables for one conceptual decision: evaluate candidate depths 14, 13, 12, and 11. The useful information is row-shaped data:

- depth / level count
- input frontier entries
- closed/rejected compensation counts
- final frontier entries
- topology pattern count
- feasible frontier entries
- failure reason if any
- whether selected

The default report should emit this as a single depth-search table.

### Context repeated across HTree sections

The same scope fields recur in multiple HTree tables:

- `clock_name`
- `clock_net_name`
- `sink_domain`
- `stage`
- `object_name_prefix`

They appear in `HTree Characterization Grid Plan`, `HTree Characterization Overview`, and `HTree Synthesis Overview`. This is useful context, but only needs to be emitted once per HTree build scope in the default report.

### Trivial stage summaries

Many stages emit a summary table with only:

```text
status = finished
```

Examples include:

- `HTree Emit synthesis summary Summary`
- `HTree build Summary`
- `Topology Commit sink domain layout Summary`
- `Topology Commit source trunk layout Summary`
- `Evaluation Evaluate CTS clock tree Summary`
- `CTS Clock tree synthesis API flow Summary`

These tables add noise because the final status and runtime tables already summarize run success. They should be suppressed by default unless the stage failed, skipped, or contributed non-trivial metrics.

### Report talks about deduplication

Some rows expose implementation commentary such as `deduplicated` rather than domain facts. Example: `CharBuilder Setup` has rows like `max_slew_source = runtime_config`, `Source = deduplicated`, `Detail = configured limit is reported in Runtime Configuration`.

This solves a previous duplication problem, but it leaves meta-reporting in the user-facing report. The final report should avoid explaining report mechanics and instead show either the resolved value, the source, or a pointer to the owning section.

### Debug and summary data share one channel

The current default `cts.log` includes both:

- high-signal user-facing summaries (`CTS Key Results`, `CTS Evaluation Overview`)
- deep algorithm trace data (`boundary_closed_candidates`, `direct_cache_hits`, `load_resolution_cache_hits`, per-depth compensation internals)

Both are valuable, but not at the same default verbosity. The lack of a report profile forces every engineer to read a debug trace even when they only need run outcome and selected CTS decisions.

### Table formatting amplifies long fields

The ASCII formatter pads every row to the widest cell in each column. One very long `Detail` value widens every row in that table. This makes `HTree Synthesis Overview` and final evaluation tables difficult to read in terminals and reviews.

## Recommended Direction

The core fix should not be a one-off deletion of tables. The report needs a structural distinction between:

- default report sections
- stage lifecycle tracing
- verbose/debug algorithm detail
- generated detail artifacts

User-confirmed product decision:

- Default `cts.log` must be a concise, high-density engineering report.
- Old repeated, redundant, and useless content must be cleaned from default output.
- More detailed but still useful internals must be written to `cts_detail.log`.
- `cts_detail.log` should not be a blind copy of the current noisy report.

The default `cts.log` should become an outcome-first report:

1. `Run Summary`
2. `Runtime Setup`
3. `Input Clocks`
4. `Synthesis Summary`
5. `HTree Decision Summary`
6. `Source Trunk Summary`
7. `Instantiation Summary`
8. `Evaluation Summary`
9. `Runtime`
10. `Artifacts`
11. `Diagnostics`

Detailed per-candidate and per-level data should move to optional verbose output or separate files referenced from the artifact section.

Updated duplicate key/value scan of the default report found only table-shape false positives:

- `clock`: appears as the first column value in multi-column clock tables (`Clock Trace Overview`, `SDC Clock Ownership Overview`, `Clock Distribution Overview`, `Source Trunk Summary`, `CTS Clock Tree Sink Domains`). These are not duplicate key/value fields; they are row identifiers in different row-shaped tables.
- `status`: appears in four true summary rows: `Runtime Routing / Wire RC`, `ReadData Overview`, `CTS Clock Tree Synthesis Overview`, and `CTS Instantiation Overview`. These are top-level stage outcomes and are intentionally retained.

Field-level cleanup applied:

- `clock_name`, `clock_net_name`, `sink_domain`, `stage`, `object_name_prefix`
  - Before: repeated across HTree grid, characterization overview, synthesis overview, and source-trunk key/value summaries.
  - Source adjustment: emit one `HTree Build Scope` table to both reports; remove repeated scope rows from HTree grid and default synthesis overview; convert source trunk to a row-shaped `Source Trunk Summary` instead of repeating scope as key/value rows.

- `resolved_wirelength_unit`
  - Before: repeated by `HTree Characterization Grid Plan` and `CharBuilder Setup` when HTree adapted the grid.
  - Source adjustment: when HTree grid adaptation supplies explicit direct wirelength indices, default `CharBuilder Setup` omits `resolved_wirelength_unit`; `HTree Characterization Grid Plan` owns that value. Non-HTree/direct CharBuilder fallback still reports the resolved unit in `CharBuilder Setup`.

- `max_slew`, `max_cap`, `slew_steps`, `cap_steps`
  - Before: repeated between runtime configuration and CharBuilder setup.
  - Source adjustment: default `CharBuilder Setup` keeps only `wirelength_points`, `routing_rc`, and `buffer_count` for adapted HTree runs; full setup rows remain in `CharBuilder Setup Detail`.

- `max_observed_output_slew_idx`, `max_observed_driven_cap_idx`, `slew_lattice_source`, `cap_lattice_source`
  - Before: default `CharBuilder Results` mixed engineering results with bucket/source trace fields.
  - Source adjustment: default `CharBuilder Results` keeps sample counts, overflow ratios, and observed max values; bucket indices and lattice source rows move to `CharBuilder Results Detail`.

- `CharBuilder Sweep Progress`
  - Before: wide per-wirelength sweep table in default `cts.log`.
  - Source adjustment: moved to `CharBuilder Sweep Progress Detail` in `cts_detail.log`; default retains aggregate `CharBuilder Results`.

- `HTree Characterization Overview`
  - Before: default table mostly pointed to other report sections (`CharBuilder Setup`, `CharBuilder Results`, grid-plan source).
  - Source adjustment: moved to `HTree Characterization Detail`; default keeps the actual owner tables instead of a meta-report table.

- `used_boundary_fallback`, `boundary_fallback_reason`
  - Before: repeated as `false` / `none` in HTree and source-trunk default summaries.
  - Source adjustment: source-trunk default summary only adds a fallback reason column when fallback is actually used; full fallback fields remain in `SourceTrunkSegment Build Detail`. HTree keeps fallback state in the selected HTree summary because it changes topology-selection interpretation.

- `inserted_insts`, `inserted_nets`
  - Before: generic names appeared in HTree summary and source-trunk summary.
  - Source adjustment: HTree selected topology keeps generic inserted counts for the downstream tree; source-trunk detail uses `segment_inserted_insts` / `segment_inserted_nets`, and the default `Source Trunk Summary` reports them as row-shaped columns.
