# Optimize CTS report log structure

## Goal

Reduce redundancy in the iCTS structured report log (`cts.log`) while preserving the information engineers need to understand CTS run quality and decisions. The default report must be a dense engineering summary, not a raw execution trace. Detailed but still curated CTS internals must move to `cts_detail.log`.

## Requirements

- The default `cts.log` generated by:
  `cd /home/liweiguo/project/ecc-tools/scripts/design/ics55_huge_dev && ./iEDA -script ./script/iCTS_script/run_iCTS_dev.tcl`
  must be shorter, easier to scan, and organized around report sections that match engineer questions:
  run result, input clocks, synthesis decisions, evaluation QoR, runtime, artifacts, and diagnostics.
- The default `cts.log` must have higher information density than the current report. Do not keep old low-value content simply for compatibility: repeated rows, duplicated context, `status=finished`-only helper summaries, report-mechanics wording, and debug counters that do not affect default engineering decisions must be removed from the default report.
- Preserve existing key engineering facts currently present in `scripts/design/ics55_huge_dev/result/cts/cts.log`, including clock ownership, selected H-tree depth/topology, characterization setup/result, source trunk result, final evaluation, generated artifacts, and runtime. Preserve them in concise form; long explanatory details may move to `cts_detail.log`.
- Collapse repeated per-depth HTree search output into one compact table in the default report. The default log must not emit the same `HTreeDepth Build pattern frontier`, `HTreeDepth Apply root-driver compensation`, and `HTreeDepth Filter sink-load region` Context/Summary table pairs once per evaluated depth.
- Generate `result/cts/cts_detail.log` beside `result/cts/cts.log`. Detailed candidate, compensation, per-level, bucket, cache, and stage lifecycle data that remains useful for CTS debugging belongs in `cts_detail.log`, not in the default `cts.log`.
- `cts_detail.log` must be curated, not a blind copy of the old noisy default log. It may contain deeper internal tables, but repeated, redundant, and useless information should still be removed or aggregated.
- Avoid repeating scope context (`clock_name`, `clock_net_name`, `sink_domain`, `stage`, `object_name_prefix`) in every HTree table when the context is unchanged. Emit the scope once, then only stage-specific values.
- Keep runtime logging and structured file reporting aligned with `.trellis/spec/backend/logging-guidelines.md`: runtime diagnostics use `LOG_*`; structured file output uses schema/report helpers.
- Keep report field ownership close to the data owner. Flow/API entry layers coordinate report timing and high-level sections; low-level modules own their field assembly.
- Preserve failure diagnostics and fallback explanations. Concision must not hide errors, warnings, fallback reasons, or first-failure information.
- Add focused tests or regression checks for the new report behavior, including repeated-stage collapse and retention of key result sections.

## Acceptance Criteria

- [ ] Running the CTS dev script produces a default `result/cts/cts.log` that starts with the high-signal run summary and can be scanned without reading per-helper Context/Summary noise.
- [ ] Running the CTS dev script also produces `result/cts/cts_detail.log` for curated deep CTS internals.
- [ ] The default log contains no repeated per-depth `HTreeDepth ... Context` / `HTreeDepth ... Summary` table pairs for the normal native H-tree search path; depth candidates are represented as rows in a single summary table.
- [ ] The default log does not contain trivial helper-stage tables whose only meaningful payload is `status=finished`.
- [ ] The default log does not contain report-mechanics wording such as `deduplicated` as user-facing content.
- [ ] The default log still includes, at minimum: run context, runtime setup, clock trace/ownership overview, clock distribution, selected H-tree topology summary, source trunk summary, synthesis flow status, instantiation overview, evaluation QoR, runtime summary, key results, report mode, and generated artifact summaries.
- [ ] Detailed HTree candidate/debug data remains accessible through `cts_detail.log`.
- [ ] `cts_detail.log` keeps useful deep-debug details such as per-depth search, compensation counters, bucket details, per-level selected H-tree data, and stage lifecycle summaries, but avoids exact repetition of unchanged scope context.
- [ ] Existing failure/no-op paths continue to emit actionable diagnostics and final status.
- [ ] Tests or scriptable assertions cover report structure changes and prevent the old redundant table pattern from returning.
- [ ] Final implementation passes the normal iCTS validation workflow required by the project specs.
