### Entry Points and Phase Order

The public synthesis entry is `Synthesis::run(ClockLayout&)`, which resets the layout, gets all design clocks, and calls `synthesizeClock` per clock (`src/operation/iCTS/source/flow/synthesis/Synthesis.cc:189`). Per clock, it clears prior topology, partitions sinks into hard-macro and regular domains, creates one `CharacterizationLibrary`, prepares non-empty sink-domain contexts, then calls `Topology::formClock` (`src/operation/iCTS/source/flow/synthesis/Synthesis.cc:91`, `src/operation/iCTS/source/flow/synthesis/Synthesis.cc:117`, `src/operation/iCTS/source/flow/synthesis/Synthesis.cc:129`, `src/operation/iCTS/source/flow/synthesis/Synthesis.cc:154`).

`Topology::formClock` first synthesizes each sink domain, then synthesizes the source-to-root trunk (`src/operation/iCTS/source/flow/synthesis/topology/Topology.cc:159`). Sink domains with fewer than two sinks are recorded as direct; otherwise `BuildSinkTree` builds the downstream H-tree and commits inserted objects (`src/operation/iCTS/source/flow/synthesis/topology/Topology.cc:198`, `src/operation/iCTS/source/flow/synthesis/topology/Topology.cc:201`, `src/operation/iCTS/source/flow/synthesis/topology/Topology.cc:207`, `src/operation/iCTS/source/flow/synthesis/topology/Topology.cc:217`). Source trunk dispatch uses a single segment when there is one root input and a top H-tree when there are multiple root inputs (`src/operation/iCTS/source/flow/synthesis/topology/trunk/SourceTrunk.cc:128`, `src/operation/iCTS/source/flow/synthesis/topology/trunk/SourceTrunk.cc:133`, `src/operation/iCTS/source/flow/synthesis/topology/trunk/SourceTrunk.cc:154`).

`HTree::build` has the native H-tree phase order:

1. Validate root driver and loads (`src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:213`, `src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:219`, `src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:227`).
2. Build a geometric binary topology with `TopologyGen::build` (`src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:236`, `src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:240`).
3. Ensure characterization and capture char grid metadata in `BuildResult` (`src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:260`, `src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:263`).
4. Resolve boundary constraints and level plans, then resolve depth candidates (`src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:271`, `src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:278`, `src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:286`).
5. Load `CharBuilder` buffering patterns into a `BufferPatternLibrary`, then synthesize segment frontiers for required aligned lengths (`src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:294`, `src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:299`, `src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:309`).
6. Build root-driver compensation options and fanout pruning options (`src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:323`, `src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:333`).
7. Search all topology depth candidates (`src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:338`, `src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:345`).
8. Filter global feasible/candidate pools by physical sink-load-region coverage (`src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:361`, `src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:367`).
9. Select the global topology candidate, preferring strict feasible entries and falling back to candidate entries only when needed (`src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:387`, `src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:393`, `src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:396`, `src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:413`).
10. Re-check selected sink-load legality, evaluate selected root-driver compensation, materialize the selected topology pattern, annotate level plans, and prevalidate root-driver sizing (`src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:431`, `src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:463`, `src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:516`, `src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:519`).
11. Build CTS object embedding, optionally apply root-driver sizing, emit summary, and return the `BuildResult` (`src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:525`, `src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:531`, `src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:534`, `src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:551`).

### Key Classes and Data Structures

`HTree::BuildOptions` carries optional branch-buffer forcing, min top input slew, target depth, depth search window, topology tolerance, fixed topology root location, shared characterization library, extra characterization lengths, root-driver sizing flag, local-buffer load semantics, clock period, log context, and object name prefix (`src/operation/iCTS/source/flow/synthesis/htree/HTree.hh:57`). `HTree::BuildResult` contains success/failure state, the generated `Tree`, selected levels, selected `HTreeTopologyChar` / `HTreeTopologyPattern`, characterization metadata, depth/selection counters, load stats, root-driver compensation report, fallback diagnostics, owned inserted `Inst` / `Pin` / `Net` objects, per-object topology levels, and root pin/net pointers (`src/operation/iCTS/source/flow/synthesis/htree/HTree.hh:139`).

`Tree` owns `TreeNode` objects. Each `TreeNode` stores parent/children IDs, a DBU position, and borrowed load pins; `Tree::levels()` returns breadth-first node IDs by level (`src/operation/iCTS/source/database/spatial/Tree.hh:37`, `src/operation/iCTS/source/database/spatial/Tree.hh:50`, `src/operation/iCTS/source/database/spatial/Tree.hh:80`, `src/operation/iCTS/source/database/spatial/Tree.hh:142`).

`HTree::LevelPlan` stores per-level requested/aligned length, leaf-level flag, selected segment pattern ID, selected buffer master/count/area data, and weighted physical counts/areas (`src/operation/iCTS/source/flow/synthesis/htree/HTree.hh:75`). `BuildLevelPlans` computes average parent-child Manhattan length per topology level and aligns it to a characterization length index (`src/operation/iCTS/source/flow/synthesis/htree/plan/Plan.cc:52`, `src/operation/iCTS/source/flow/synthesis/htree/plan/Plan.cc:61`, `src/operation/iCTS/source/flow/synthesis/htree/plan/Plan.cc:84`, `src/operation/iCTS/source/flow/synthesis/htree/plan/Plan.cc:88`).

`CharacterizationLibrary` wraps `CharBuilder`, caches the last `InitOptions` request, and reuses the built segment chars when the key matches (`src/operation/iCTS/source/flow/synthesis/htree/characterization/library/CharacterizationLibrary.hh:34`, `src/operation/iCTS/source/flow/synthesis/htree/characterization/library/CharacterizationLibrary.cc:36`, `src/operation/iCTS/source/flow/synthesis/htree/characterization/library/CharacterizationLibrary.cc:53`). The shared database/module characterization types are `SegmentChar`, `HTreeTopologyChar`, `BufferingPattern`, `HTreeTopologyPattern`, `PatternId`, and `UniformValueLattice`.

`BufferPatternLibrary` stores segment `BufferingPattern`s by `PatternId` and caches `PatternCompositionState`, including terminal semantic, monotonic boundary state, and exposed load count (`src/operation/iCTS/source/flow/synthesis/htree/segment_pruning/SegmentLibrary.hh:116`). `SegmentFrontierCatalog` maps aligned length index to a `SegmentCandidateFrontierSet`, which can contain `kAll`, `kTerminalBranchBuffered`, and `kTerminalLeafUnbuffered` frontiers (`src/operation/iCTS/source/flow/synthesis/htree/segment_pruning/SegmentLibrary.hh:46`, `src/operation/iCTS/source/flow/synthesis/htree/segment_pruning/SegmentLibrary.hh:200`, `src/operation/iCTS/source/flow/synthesis/htree/segment_pruning/SegmentLibrary.hh:268`).

`TopologyPatternLibrary` stores seed/concat nodes for topology patterns and can materialize a selected topology pattern into the sequence of root-to-leaf segment pattern IDs (`src/operation/iCTS/source/flow/synthesis/htree/segment_pruning/SegmentLibrary.hh:336`, `src/operation/iCTS/source/flow/synthesis/htree/segment_pruning/SegmentLibrary.hh:355`, `src/operation/iCTS/source/flow/synthesis/htree/segment_pruning/SegmentLibrary.hh:410`). `CandidateBuildEvaluation` records a candidate depth's levels, pattern library, final/candidate/feasible frontiers, selected char, and fallback metadata (`src/operation/iCTS/source/flow/synthesis/htree/topology_pruning/TopologyPruning.hh:45`). `DepthSearchResult` holds all per-depth evaluations plus global feasible/candidate ref pools and shared sink-load/root-driver compensation context (`src/operation/iCTS/source/flow/synthesis/htree/plan/DepthPlan.hh:69`).

### How Topology Is Chosen

`TopologyGen` builds a full binary tree with a power-of-two leaf count. Without explicit target depth, leaf count is the largest power of two not exceeding load count; target depth clamps leaf count to `1 << depth` (`src/operation/iCTS/source/module/topology/TopologyGen.cc:197`, `src/operation/iCTS/source/module/topology/TopologyGen.cc:357`, `src/operation/iCTS/source/module/topology/TopologyGen.cc:366`). The root location is either fixed by options or set to the load median (`src/operation/iCTS/source/module/topology/TopologyGen.cc:213`, `src/operation/iCTS/source/module/topology/TopologyGen.cc:215`). `buildFullTree` creates the binary shape (`src/operation/iCTS/source/module/topology/TopologyGen.cc:378`). `embedPositions` recursively bi-partitions loads with `Clustering::biPartition`, assigns child centers, and stores each node's load subset (`src/operation/iCTS/source/module/topology/TopologyGen.cc:398`, `src/operation/iCTS/source/module/topology/TopologyGen.cc:419`, `src/operation/iCTS/source/module/topology/TopologyGen.cc:430`, `src/operation/iCTS/source/module/topology/TopologyGen.cc:433`, `src/operation/iCTS/source/module/topology/TopologyGen.cc:450`). `balanceTopology` then constrains each level's parent-child Manhattan length toward the level average, within `htree_topology_tolerance` and load bounds (`src/operation/iCTS/source/module/topology/TopologyGen.cc:460`, `src/operation/iCTS/source/module/topology/TopologyGen.cc:487`, `src/operation/iCTS/source/module/topology/TopologyGen.cc:503`).

After the geometric topology is built, H-tree synthesis may evaluate multiple candidate depths. `ResolveDepthCandidates` returns an explicit clamped target depth when supplied; otherwise it evaluates a descending window from the maximum topology depth, using `CONFIG_INST.get_htree_depth_explore_window()` unless overridden (`src/operation/iCTS/source/flow/synthesis/htree/plan/Plan.cc:138`, `src/operation/iCTS/source/flow/synthesis/htree/plan/Plan.cc:144`, `src/operation/iCTS/source/flow/synthesis/htree/plan/Plan.cc:149`, `src/operation/iCTS/source/flow/synthesis/htree/plan/Plan.cc:154`). Each candidate depth truncates the level plan and treats that depth's topology nodes as leaves (`src/operation/iCTS/source/flow/synthesis/htree/plan/DepthPlan.cc:42`, `src/operation/iCTS/source/flow/synthesis/htree/plan/Plan.cc:106`, `src/operation/iCTS/source/flow/synthesis/htree/plan/Plan.cc:129`).

For each depth, `BuildPatternSearch` walks levels from leaf to root, creates topology seed entries from segment frontiers, composes upstream/downstream frontiers with hash-join concatenation, applies root-driver compensation, state-frontier pruning, and root fanout filtering (`src/operation/iCTS/source/flow/synthesis/htree/topology_pruning/TopologyPruning.cc:230`, `src/operation/iCTS/source/flow/synthesis/htree/topology_pruning/TopologyPruning.cc:245`, `src/operation/iCTS/source/flow/synthesis/htree/topology_pruning/TopologyPruning.cc:272`, `src/operation/iCTS/source/flow/synthesis/htree/topology_pruning/TopologyPruning.cc:279`, `src/operation/iCTS/source/flow/synthesis/htree/topology_pruning/TopologyPruning.cc:294`, `src/operation/iCTS/source/flow/synthesis/htree/topology_pruning/TopologyPruning.cc:309`, `src/operation/iCTS/source/flow/synthesis/htree/topology_pruning/TopologyPruning.cc:313`). `SearchTopologyDepthCandidates` records each evaluation and appends its feasible/candidate frontier refs into global pools (`src/operation/iCTS/source/flow/synthesis/htree/plan/DepthPlan.cc:101`, `src/operation/iCTS/source/flow/synthesis/htree/plan/DepthPlan.cc:118`, `src/operation/iCTS/source/flow/synthesis/htree/plan/DepthPlan.cc:125`).

Global selection filters both strict-feasible and fallback candidate refs by sink-load-region coverage, compresses each depth by delay/power Pareto refs, then chooses `SelectBestGlobalEntry`. The best global entry is the lower median of the delay/power Pareto front after sorting by power-first tie-breakers (`src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:367`, `src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:393`, `src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:397`, `src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:399`, `src/operation/iCTS/source/flow/synthesis/htree/topology_pruning/TopologyPruning.cc:434`, `src/operation/iCTS/source/flow/synthesis/htree/topology_pruning/TopologyPruning.cc:460`, `src/operation/iCTS/source/flow/synthesis/htree/topology_pruning/TopologyPruning.cc:466`, `src/operation/iCTS/source/flow/synthesis/htree/topology_pruning/TopologyPruning.cc:471`). The same median-of-Pareto policy is used for per-depth `best_char` (`src/operation/iCTS/source/flow/synthesis/htree/topology_pruning/TopologyPruning.cc:346`, `src/operation/iCTS/source/flow/synthesis/htree/topology_pruning/TopologyPruning.cc:352`, `src/operation/iCTS/source/flow/synthesis/htree/topology_pruning/TopologyPruning.cc:362`).

### How Segments Are Chosen

`CharBuilder` supplies base segment chars and base buffering patterns. `HTree::build` inserts all `get_buffering_patterns()` into `BufferPatternLibrary` (`src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:294`). `MakeHTreeSegmentFrontierRequest` requests either all frontiers or branch-buffer-constrained frontiers depending on `force_branch_buffer` (`src/operation/iCTS/source/flow/synthesis/htree/segment_pruning/SegmentPruning.cc:405`).

`SynthesizeSegmentFrontiers` first groups base `SegmentChar`s by length and frontier kind, then builds state frontiers with composition-state-aware Pareto pruning (`src/operation/iCTS/source/flow/synthesis/htree/segment_pruning/SegmentPruning.cc:144`, `src/operation/iCTS/source/flow/synthesis/htree/segment_pruning/SegmentPruning.cc:161`, `src/operation/iCTS/source/flow/synthesis/htree/segment_pruning/SegmentPruning.cc:177`). If a required aligned length lacks base entries, it recursively splits the length into left/right sublengths, solves subproblems with memoization, composes segment frontiers via hash join, and chooses the lowest-cost feasible closure with more frontier entries as a tie-breaker (`src/operation/iCTS/source/flow/synthesis/htree/segment_pruning/SegmentPruning.cc:268`, `src/operation/iCTS/source/flow/synthesis/htree/segment_pruning/SegmentPruning.cc:291`, `src/operation/iCTS/source/flow/synthesis/htree/segment_pruning/SegmentPruning.cc:313`, `src/operation/iCTS/source/flow/synthesis/htree/segment_pruning/SegmentPruning.cc:339`, `src/operation/iCTS/source/flow/synthesis/htree/segment_pruning/SegmentPruning.cc:347`, `src/operation/iCTS/source/flow/synthesis/htree/segment_pruning/SegmentPruning.cc:363`). Segment composition requires monotonic buffering compatibility through `BufferingPattern::canConcatMonotonic` (`src/operation/iCTS/source/flow/synthesis/htree/segment_pruning/SegmentLibrary.hh:470`, `src/operation/iCTS/source/flow/synthesis/htree/segment_pruning/SegmentLibrary.hh:479`).

Single source-to-root segment synthesis follows a narrower variant: ensure characterization, align the physical source-to-root length to a `length_idx`, filter the segment frontier by required load cap, source drive cap, and optional min input slew, then select the median of a delay/power Pareto front (`src/operation/iCTS/source/flow/synthesis/topology/trunk/SourceTrunkSegment.cc:352`, `src/operation/iCTS/source/flow/synthesis/topology/trunk/SourceTrunkSegment.cc:385`, `src/operation/iCTS/source/flow/synthesis/topology/trunk/SourceTrunkSegment.cc:412`, `src/operation/iCTS/source/flow/synthesis/topology/trunk/SourceTrunkSegment.cc:430`, `src/operation/iCTS/source/flow/synthesis/topology/trunk/SourceTrunkSegment.cc:461`, `src/operation/iCTS/source/flow/synthesis/topology/trunk/SourceTrunkSegment.cc:469`, `src/operation/iCTS/source/flow/synthesis/topology/trunk/SourceTrunkSegment.cc:472`).

### How Characterization Is Used

`CharacterizationLibrary::buildRuntimeOptions` reads config values for max slew/cap, wirelength unit/iterations, slew/cap steps, buffer types, redundancy, routing layer, and wire width (`src/operation/iCTS/source/flow/synthesis/htree/characterization/library/CharacterizationLibrary.cc:74`). `RunCharacterizationFlow` collects requested topology level lengths and caller-supplied additional source-to-root lengths, resolves an adapted grid when runtime config is missing or collapses distinct level bins, optionally passes direct length indices to `CharBuilder`, then calls `CharacterizationLibrary::ensure` (`src/operation/iCTS/source/flow/synthesis/htree/characterization/Characterization.cc:81`, `src/operation/iCTS/source/flow/synthesis/htree/characterization/Characterization.cc:83`, `src/operation/iCTS/source/flow/synthesis/htree/characterization/Characterization.cc:128`, `src/operation/iCTS/source/flow/synthesis/htree/characterization/Characterization.cc:137`). It fails if the `CharBuilder` has no segment chars or no positive wirelength unit, then copies characterization metadata into `HTree::BuildResult` (`src/operation/iCTS/source/flow/synthesis/htree/characterization/Characterization.cc:147`, `src/operation/iCTS/source/flow/synthesis/htree/characterization/Characterization.cc:153`).

The grid adaptation rule is in `ResolveCharacterizationGridPlan`: if configured unit is missing or multiple requested lengths collapse into one aligned bin, it derives a fallback unit as `max_requested_length / requested_length_count`, computes required covering iterations, and caps direct characterization bins by configured iterations (`src/operation/iCTS/source/flow/synthesis/htree/characterization/wirelength/WirelengthGrid.cc:139`, `src/operation/iCTS/source/flow/synthesis/htree/characterization/wirelength/WirelengthGrid.cc:147`, `src/operation/iCTS/source/flow/synthesis/htree/characterization/wirelength/WirelengthGrid.cc:158`, `src/operation/iCTS/source/flow/synthesis/htree/characterization/wirelength/WirelengthGrid.cc:162`, `src/operation/iCTS/source/flow/synthesis/htree/characterization/wirelength/WirelengthGrid.cc:174`). Direct length indices are dense when required iterations exceed the cap, otherwise unique covering indices for requested lengths (`src/operation/iCTS/source/flow/synthesis/htree/characterization/wirelength/WirelengthGrid.cc:191`, `src/operation/iCTS/source/flow/synthesis/htree/characterization/wirelength/WirelengthGrid.cc:198`, `src/operation/iCTS/source/flow/synthesis/htree/characterization/wirelength/WirelengthGrid.cc:202`).

Boundary constraints map `min_top_input_slew_ns` to a covering slew lattice index; if root-driver sizing is enabled, `HTree::build` removes the top input slew constraint from the pattern search and instead requires root-driver compensation to close the physical boundary (`src/operation/iCTS/source/flow/synthesis/htree/constraint/Constraint.cc:45`, `src/operation/iCTS/source/flow/synthesis/htree/constraint/Constraint.cc:50`, `src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:124`, `src/operation/iCTS/source/flow/synthesis/htree/HTree.cc:275`). Candidate filtering still handles explicit boundary constraints when present (`src/operation/iCTS/source/flow/synthesis/htree/topology_pruning/TopologyPruning.cc:327`, `src/operation/iCTS/source/flow/synthesis/htree/topology_pruning/TopologyPruning.cc:501`).

`SinkLoadRegion` uses the cap lattice from characterization to prove that the selected bottom-most buffered boundary can legally drive real external load groups and that the selected char's leaf-load cap bucket covers the physical group cap (`src/operation/iCTS/source/flow/synthesis/htree/region/SinkLoadRegion.cc:252`, `src/operation/iCTS/source/flow/synthesis/htree/region/SinkLoadRegion.cc:267`, `src/operation/iCTS/source/flow/synthesis/htree/region/SinkLoadRegion.cc:316`, `src/operation/iCTS/source/flow/synthesis/htree/region/SinkLoadRegion.cc:345`, `src/operation/iCTS/source/flow/synthesis/htree/region/SinkLoadRegion.cc:347`, `src/operation/iCTS/source/flow/synthesis/htree/topology_pruning/TopologyPruning.cc:597`).
